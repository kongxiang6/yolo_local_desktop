# YOLO 训练工具

一个面向普通用户的本地桌面版 YOLO 工具，尽量把 **环境配置、数据整理、训练、验证、预测、跟踪、导出** 全部收进一个界面里完成。

> 适合给自己用，也适合打包后直接发给同事、客户、朋友使用。  
> 目标不是“会命令的人更快”，而是“不会命令的人也能顺利用起来”。

## 项目简介

很多 YOLO 工具本身并不难，难的是：

- 第一次装环境容易报错
- 参数一多，新手不知道该怎么填
- 打包给别人之后，别人电脑上环境不一定能跑
- 很多人只会双击 `exe`，不会看命令行，也不会修 Python

这个项目就是为了解决这些问题：

1. 尽量把环境配置自动化  
2. 尽量把常用流程做成界面操作  
3. 尽量把参数提示写成小白能看懂的语言  
4. 尽量减少弹出多余黑框、命令行窗口  

---

## 核心特性

### 1. 一体化工作流

软件当前已经把这几条常用链路整合进一个桌面界面：

- 运行环境检查
- 在线一键配置环境
- 原始检测数据整理
- 模型训练
- 模型验证
- 图片 / 视频 / 摄像头 / RTSP / HTTP 流预测
- 目标跟踪
- 模型导出

### 2. 面向小白的参数说明

所有主要参数都支持提示说明，尽量包含：

- 参数是做什么的
- 默认值是什么
- 类型是什么
- 新手一般怎么填
- 常见示例怎么写

### 3. 可分发的内置运行环境逻辑

软件支持：

- 优先使用内置 `runtime`
- 内置环境缺失时自动创建
- 环境不完整时自动修复 / 更新
- 检查“当前解释器”是否真的可用
- 避免把主程序 `exe` 误识别成 Python

### 4. 更适合交付他人的使用方式

为了方便发给别人用，当前版本重点照顾了这些场景：

- 对方没装 Python
- 对方不会装 `pip`
- 对方不会装 `torch`
- 对方不会看命令行
- 对方需要直接打开软件就能跑

## 当前功能核查结果

我已经按当前项目代码和界面做过功能核查，当前版本 **没有发现主流程缺失项**。

### 已具备的完整闭环

- 没环境时：可以自动创建内置环境
- 环境损坏时：可以自动检查并修复
- 有原始标注时：可以先整理数据集
- 有模型 / 权重时：可以训练、验证、预测、跟踪、导出
- 不懂参数时：可以看提示或直接套用推荐预设

### 当前结论

如果目标是“把软件打包后给很多人直接用”，那么最关键的闭环已经具备：

1. 环境自动处理  
2. 主流程界面化  
3. 参数提示可读化  
4. 结果和日志可追踪  

也就是说，当前版本已经能作为一套 **可分发、对新手友好、能直接跑主流程** 的桌面工具使用。

## 运行环境逻辑

这是整个软件最关键的设计之一，因为它直接决定软件能不能发给别人就用。

### 默认逻辑

- 默认优先使用软件目录里的内置 `runtime`
- 如果内置环境完整，直接使用
- 如果内置环境缺失或损坏，自动进入在线配置流程

### 点击“内置”时

- 如果已经检测到完整环境：直接切换到内置环境
- 如果没有检测到完整环境：自动触发“在线一键配置环境”

### 点击“当前”时

- 会先检查当前选择的解释器是不是真正的 Python
- 不会再把主程序 `exe` 错当成解释器

### 点击“在线一键配置环境”时

- 只会检查和配置软件内置 `runtime`
- 如果内置环境已经可用，会先询问是否继续更新
- 如果内置环境不可用或不完整，才继续安装 / 修复
- 如果软件目录里没有 `runtime`，会先自动创建内置 Python 再继续安装依赖

### 当前环境安装流程

配置过程中会尽量自动处理：

- Python
- `pip`
- `torch`
- `ultralytics`
- `pillow`
- `pyyaml`

### 为什么这样设计

这样设计主要是为了：

- 降低小白第一次使用门槛
- 让软件更适合直接分发
- 减少“别人电脑上不能跑”的情况
- 避免用户手动修命令、修路径、修环境

### 提示会显示在哪里

软件里主要有两种提示方式：

1. 参数下方直接显示说明文字
2. 鼠标停在参数名或输入框上时，弹出提示

### 提示会尽量包含什么内容

- 参数用途

- 默认值

- 新手建议

## 常用参数解释

下面是最常用的一批参数，用更通俗的话解释。

### 训练常用参数

#### `epochs`

训练轮数，可以理解成“把这批数据反复学习多少遍”。

#### `batch`

每次同时送进显卡 / CPU 的图片数量。  
太大容易爆显存；显存不够时，优先减小它。

#### `imgsz`

输入尺寸。  
一般越大越清楚，但更慢、更吃显存。

#### `device`

用什么设备运行：

- `0`：第一张显卡
- `0,1`：多卡
- `cpu`：只用 CPU

#### `workers`

数据读取线程数。  
如果电脑比较卡、硬盘比较慢，可以适当调小。

#### `patience`

多久没有变好就提前停止训练，避免白跑很久。

#### `amp`

自动混合精度。  
大多数显卡建议开启，通常更快、更省显存。

### 验证 / 预测 / 跟踪常用参数

#### `conf`

置信度阈值。  
越高越严格，误检会少一些，但也可能漏检。

#### `iou`

去重阈值。  
一般保持默认即可。

#### `max_det`

一张图最多保留多少个目标。

#### `save`

是否保存结果图或结果视频。

#### `save_txt`

是否额外把结果保存成 `txt`。

#### `save_crop`

是否把识别到的目标单独裁剪保存。

#### `tracker`

跟踪算法选择。  
不确定时保持默认即可。

### 导出常用参数

#### `format`

导出格式。  
如果不确定部署环境，通常优先试 `ONNX`。

#### `dynamic`

是否支持动态输入尺寸。  
更灵活，但某些平台兼容性可能稍差。

#### `half`

是否使用半精度。  
通常更省显存、更快，但要看平台支不支持。

#### `int8`

更激进的量化压缩。  
模型会更小、更快，但不一定适合所有部署环境。

## 常见问题

### 没装 Python 能用吗？

可以。  
软件支持自动创建内置 Python 运行环境。

### 点“一键配置环境”是不是每次都重装？

不是。  
现在会先检查环境，再决定要不要更新或修复。

### 为什么不要只发一个 `exe`？

因为运行环境、资源文件、预设和说明文件都可能依赖同一目录结构。  
只发一个 `exe`，别人电脑上很容易缺文件、缺环境或运行异常。

### 数据不是标准 YOLO 格式怎么办？

先用“数据整理”功能转成标准数据集，再开始训练。

### 训练好的模型怎么部署？

到“导出工作台”里导出成 ONNX、TensorRT、OpenVINO 等格式。

### 出错了怎么办？

建议优先：

1. 看左侧实时日志
2. 点“打开日志”
3. 把报错截图或日志发回来
